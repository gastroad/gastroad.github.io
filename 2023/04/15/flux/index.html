flux 패턴이란

오늘은 flux 패턴에 대해서 알아보려고 한다
react 를 사용해서 개발을 진행하게 되면 한번쯤은 들어봤을거라고 생각한다.

본론으로 들어가기전 이 포스트는 개인의 경험과 이해를 바탕으로 쓰여졌으므로
틀린 내용이 있을수 있습니다.
혹시 틀린 내용을 발견하게 된다면 gostroad37@gmail.com 으로 메일 주시면 감사하겠습니다.
정확한 내용은 하단 참고 링크에 쓰여진 공식 document를 참고하기를 바랍니다.

FLUX 란

Application architecture for building user interfaces

즉 사용자가 보는 화면을 만들기 위한 어플리케이션 구조 라고 이해했다.

단방향 데이터 흐름을 활용해 뷰 컴포넌트를 구성하는 React를 보완하기 위해서 개발된 것으로 알고있다.

바로 flux 의 구조와 데이터 흐름에 대해서 알아보도록 하자

공식 도큐먼트를 보면 아래와 같이 설명한다

All data flows through the dispatcher as a central hub. Actions are provided to the dispatcher in an action creator method, and most often originate from user interactions with the views. The dispatcher then invokes the callbacks that the stores have registered with it, dispatching actions to all stores. Within their registered callbacks, stores respond to whichever actions are relevant to the state they maintain. The stores then emit a change event to alert the controller-views that a change to the data layer has occurred. Controller-views listen for these events and retrieve data from the stores in an event handler. The controller-views call their own setState() method, causing a re-rendering of themselves and all of their descendants in the component tree.

정리하자면
모든 action 은 dispatcher 에게 action creator 메소드를 제공한다
dispatcher는 action을 store 로 전달한다
store 를 바탕으로 view 를 랜더링한다
라고 이해를 했다.

위와 같은 데이터 흐름으로 flux 는 단방향(action => dispatcher => store => view) 데이터 흐름을 만들었다.

그러면 이쯤에서 궁금하게 된것이 react 도 결국 단방향 데이터 흐름을 가지고 있는데 왜 flux 라는것이 필요하고 왜 굳이 store 에 dispatcher 를 통해 action 을 보내는 번거로운 짓을 해야하는거지?
심지어 flux 패턴을 구현한 redux 는 어마무시한 Boilerplate 코드 때문에 한동안 사용하지않으려고 별 이상한 시도를 많이했었다. 현재는 RTK 덕분에 애용하고있다

flux 아키텍처를 개발한 이유가 mvc 패턴이 양방향 데이터 흐름을 가지고 있고
이 양방향 데이터 흐름이 시스템의 복잡도를 기하급수적으로 증가 시키고, 예측 불가능한 코드를 만들게 된다고 했다 이 예측 불가능한 코드가
예측 불가능한 버그를 쏟아져 내기 떄문에 만든 아키텍쳐가 flux 인데

react 도 결국 부모 component 에서 자식 component 로 props 를 전달하고 자식은 부모로 부터 받은 props 를 수정할수 없으니 단방향 데이터 흐름을 가지고 있는거 아닌가? 라는 생각이 들었다

하지만 위와같이 flux 패턴으로 개발하게 되면 component 의 종속성없이
단방향으로 데이터를 흐르게 할수있다
이렇게 되면 프로젝트 의 확장에 따라 부모 component 가 점점 복잡해지는 문제를 해결할수있다.

이번 포스트에서 flux 패턴을 구현하는방법이나 flux 패턴을 구현한 redux, recoil 같은 라이브러리도 다룰까 생각했으나 포스트가 너무 길어질거같아 다음 에 작성해보려한다.
