todolendar

포트폴리오 겸 공부한 기술을 실 프로젝트처럼 작성해 보고 싶어 todolendar라는 프로젝트를 기획하였다.

todo-list + calendar로 달력에 일정을 적어 둘 수 있는 프로젝트를 만들어 보았다.

주 목표는 다음과 같다.

  1. atomic 패턴 사용
  2. jest를 통한 테스트 작성
  3. storybook 작성

그리고 여유가 된다면

  1. react-native로 웹뷰 만들기
  2. react-query 사용
  3. 반응형 웹앱 개발

까지 충족하고 싶어서 프로젝트를 시작하였다.

특히 atomic 패턴의 경우 실무에서 바로 사용하게 된다면 상당히 난해할 것 같아서 꼭 미리 한번 사용해 보고 싶었다.

현재 2023-06-29 일 기준으로
주 목표 3가지는 이루었다.

atomic 패턴

atomic 패턴을 사용해서 이루고자 했었던 목표는

  1. test, story 작성을 조금 더 편하게
  2. ui 와 상태 관리 관심사 분리

두 가지를 이루고 싶어서 사용을 해 보았다.

결론부터 말하자면 생각처럼 잘 되지는 않았다.

우선 디렉토리 구조를 어떻게 만들었는지부터 설명을 하겠다.

src/component/atom

더 이상 나눌 필요가 없는 단위까지 나눈 컴포넌트를 배치해 두었다.

src/component/molecules

atom으로 두기에는 구조가 복잡하거나 다수의 atom 을 사용해서 만든 컴포넌트를 배치해 두었다.

src/component/organisms

atom, molecules를 사용해서 만든 컴포넌트를 배치해 두었다.

src/templates

위 3가지 컴포넌트들을 조합해서 화면을 구성하는 컴포넌트를 배치해 두었다.

src/pages

templates에 데이터를 주입하는 컴포넌트를 배치해 두었다.

organisms를 만들면서 가장 많은 고민을 했다.
처음에는 독립적으로 기능을 수행할 수 있는 단위를 생각하고 만들었기에
상태 관리를 organisms 단계에서부터 하기 시작했다. atomic 패턴을 잘못 이해하고 있었던 거 같다.
Calendar 작업하던 중 문득 든 생각이 organisms 단계에서 상태 관리를 진행했을 경우 재사용성에 크게 문제가 생겨 props로 넘겨받게 수정을 진행했다.

직접 atomic 패턴으로 개발을 해 보니 몇 가지 단점들이 눈에 보이기 시작했다.

  1. props drilling

atomic 패턴의 특성상 정말 많은 component를 만들어서 조합하고 사용한다.
그러다 보니 필연적으로 props drilling이 심해진다.

  1. 설정의 번거로움
    많은 component를 만들다 보니 쳐야 하는 코드 수 자체가 증가한다
    각각의 컴포넌트당 index, component, test, stories, style 등의 파일을 생성해 주어야 해서 상당히 번거로웠다.
    초기 설정의 경우 webpack, typscript 등의 alias를 설정하는 것도 상당히 번거로웠다.

눈에 띄는 장점들도 있다.

  1. storybook
    storybook 을 보니 상당히 괜찮다는 느낌을 받았다.
    atom, molecules, organisms, templates 4가지 종류의 컴포넌트에 story를 작성했다.
    작성을 다 해두고 보니 어떤컴포넌트를 어떻게 사용했는지 확실하게 보이고
    figma라도 연동시켜두면 디자이너와 협업할 때 엄청 편리할 것 같다고 생각했다.
